# Supabase schema

Apply files in `migrations/` in timestamp order. The deployed project's pre-existing core tables (`organisations`, `contacts`, `projects`, `sites`, `site_assessments`, `assessment_notes`, `uploaded_files`, and `status_history`) remain a prerequisite.

The production-readiness migrations add Supabase Auth profiles, explicit organisation memberships, server-enforced role policies, persistent assignment/SLA history, a private evidence bucket, PostGIS utility territories, analyst-resolved inference suggestions, and one GIN-backed portal search index.

New auth users default to `customer`. Self-service customers are provisioned into one organisation through `provision_customer_account`; administrator invitations may supply trusted `organisation_id` and `role` values in Auth app metadata. Tenant authorisation uses `organisation_memberships`, while `profiles.organisation_id` remains a compatibility pointer to the active organisation. Populate `utility_service_territories` from authoritative provider polygons before treating geospatial inference as more than a suggestion.

Run `tests/customer_tenancy_rls.sql` against an isolated migrated Supabase database to exercise the allowed customer golden-path inserts and a denied cross-tenant insert. The fixture is transactional and rolls its records back.

Customer smart-intake drafts are stored in `customer_intake_drafts`. Pre-submission files use private `assessment-evidence` paths under `drafts/<user-id>/<draft-id>/`; `customer_intake_files` retains checksum, MIME, size, processing, malware-scan, and retention metadata before those files are linked to `uploaded_files` at submission.

Assessment lifecycle changes must use `transition_assessment_status`; direct status updates are rejected unless they match a role-approved transition, and every successful transition appends both status history and an `assessment_events` entry. `assessment_events` is the shared activity stream; customer reads exclude internal events through RLS.

File processing uses `background_jobs`. Service-role workers claim leased work with `claim_background_jobs(worker_id, job_types, limit)`. Uploads enqueue malware scanning first; a clean scan enqueues document extraction. Running jobs whose lease is older than 15 minutes can be reclaimed, and failed work retries with exponential backoff up to `max_attempts`.

After `20260621100000_file_processing_workers.sql`, run the file worker from `web/` with `npm run worker:files` (or add `-- --once` for a scheduled single batch). The worker downloads private evidence with the service role, sends it only to configured HTTPS malware-scanner and document-extractor adapters, persists extraction text in `uploaded_file_extractions`, and adds that text to permission-aware portal search. Configure the worker-only variables documented in `web/.env.example`; never expose the service-role key or adapter tokens through `NEXT_PUBLIC_*`. Job completion is lease-bound through `complete_claimed_background_job`, and extraction persistence is lease-bound through `save_uploaded_file_extraction`.

Adapter endpoints receive a multipart `POST` containing `file`, `job_id`, `uploaded_file_id`, `mime_type`, and `checksum_sha256`. The scanner must return JSON with `status: "clean" | "quarantined"` and may include `engine`, `engineVersion`, and `signature`. The extractor must return `text` (or `contentText`) and may include `pageCount`, `language`, `extractor`, `extractorVersion`, and a JSON `metadata` object. Adapter errors are retried through the durable queue; file bytes and extracted text are never written to worker logs.

Geospatial sources are registered in `geospatial_datasets`; only active versions participate in utility, TSP, and market-zone resolution. After applying `20260619160000_geospatial_dataset_governance.sql`, validate the PUCT endpoint with `npm run data:import:puct-territories -- --dry-run` from `web/`, then set `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` and run the command without `--dry-run`. The importer atomically replaces the registered dataset, validates its polygons in PostGIS, stores a checksum and feature count, and activates the version. The PUCT digital IOU layer is explicitly labelled approximate and suitable for screening rather than legal service-territory determination.

Readiness scoring is versioned by `analysis_methodology_versions`. After `20260619180000_scoring_confidence_verdict_history.sql`, manual components must be saved through `save_assessment_scores`; the server calculates the seven-factor weighted readiness, band, independent confidence, and blockers. Verdicts must be saved through `save_assessment_verdict` using a canonical value, supported confidence, rationale, conditions when applicable, and a reason for material changes. Score and verdict events and calculation snapshots are append-only; clients must never write the current score or verdict tables directly.

Evidence and report finalisation are controlled after `20260619200000_evidence_lineage_preflight.sql`. Findings and material report claims record supporting, contradicting, or contextual evidence; claim links retain evidence snapshots, and evidence gaps carry severity, owner, due date, impact and an explicit resolution. Run `run_assessment_preflight` for an auditable blocker report and `finalize_assessment_report` to move ready sections into final review. Direct final-section or ready-for-review mutations are rejected. Only administrators or reviewers can approve a blocker-specific exception through `approve_delivery_exception`, and every applied exception remains in the preflight snapshot.

Expert review is version-bound after `20260619230000_expert_review_version_approval.sql`. Finalising a report increments its immutable version snapshot. Use `save_expert_review_packet` for assignment, reviewer comments, checklist results, requested changes, rejection, and approval; direct review-table writes are rejected. Only administrators and reviewers can decide a review, approval requires a completed checklist without failures, and delivery preflight accepts only an approval tied to the current finalised export ID and version.

Issued report artifacts and customer delivery are controlled after `20260620000000_report_artifacts_secure_delivery.sql`. `request_report_artifact_generation` freezes the approved report version after delivery preflight; the authenticated server route generates and checksums the PDF and map, uploads them to the private `report-artifacts` bucket, and completes the version through `complete_report_artifact_generation`. `deliver_report_version` publishes only a complete approved version to the assessment organisation, while `revoke_report_delivery` disables new downloads. Download routes issue 60-second signed URLs and append immutable audit events.

Apply `20260620100000_customer_internal_visibility.sql` before exposing the customer portal. It keeps grid analysis, evidence curation, findings, scoring, expert review, preflight and draft report records internal while preserving customer access to their intake, their own files, shared activity, and explicitly delivered report artifacts. Apply `20260620110000_report_artifact_immutability.sql` with the corresponding web route: storage objects become write-once, retries use attempt-specific paths, and only the active generation token can complete or fail an issued version.

Expert-review decisions are assignment-bound after `20260620120000_expert_review_assignment_checklist.sql`: the server owns the six canonical checklist definitions, reviewers may decide only reviews assigned to them, and administrator overrides are separately audited. `20260620130000_report_revision_workflow.sql` adds `start_report_revision`, which reopens finalized sections after requested changes or rejection while preserving prior snapshots and issued artifacts. `20260620140000_upload_security_metadata.sql` makes customer upload metadata and stored objects immutable, moves draft-file linking into `link_customer_intake_files`, and denies signed reads until the processing worker records a clean malware scan.

Customer submission is atomic after `20260620150000_atomic_customer_intake_submission.sql`. `submit_customer_intake_draft` locks the draft, persists its latest payload, provisions the organisation/contact/project/site, creates the assessment, marks the draft submitted, and links pending files in one transaction. A retry returns the assessment already bound to the draft; any failed step rolls the statement back without partial submission state.
